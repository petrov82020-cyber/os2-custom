Глава 4: Пользовательский домен, темы Jekyll и автоматическая сборка

✍️ Время чтения: ~20 минут | 🎯 Цель: привязать свой домен, использовать готовые темы Jekyll и настроить автоматическую сборку через GitHub Actions

Мы уже научились создавать статические сайты на GitHub Pages. Но что, если вы хотите использовать собственный домен (например, my-site.com вместо ваше_имя.github.io)? Или вам нужен удобный способ ведения блога без ручного создания каждой страницы? Тогда пригодятся пользовательские домены, Jekyll (генератор статических сайтов) и GitHub Actions для автоматизации сборки.

💡 Зачем это нужно?
- Профессиональный вид сайта с собственным доменом.
- Возможность вести блог с постами в формате Markdown (Jekyll).
- Автоматическая сборка и тестирование при каждом push (GitHub Actions).

4.1 Подключение пользовательского домена

У вас уже есть адрес ваше_имя.github.io/название_репозитория. Чтобы привязать свой домен (например, my-site.com), нужно выполнить два действия: настройку DNS и настройку в репозитории.

Шаг 1: Настройка DNS у регистратора домена

Зайдите в панель управления вашего домена (где вы его купили). Добавьте следующие записи:

  • A-запись для корневого домена (например, @) со значением 185.199.108.153 (а также 185.199.109.153, 185.199.110.153, 185.199.111.153) – все четыре IP-адреса GitHub Pages.
  • CNAME-запись для поддомена www на ваше_имя.github.io.

Подождите от нескольких минут до нескольких часов – DNS распространяется не мгновенно.

Шаг 2: Настройка в репозитории

  1. В репозитории перейдите в Settings → Pages.
  2. В поле Custom domain введите ваш домен (например, my-site.com) и нажмите Save.
  3. GitHub автоматически создаст файл CNAME в корне вашего репозитория (содержит ваш домен). Не удаляйте его.

После этого сайт будет доступен по вашему домену.

⚠️ Важно: Если вы используете проект GitHub Pages (не пользовательский), то адрес будет ваше_имя.github.io/репозиторий, и собственный домен также можно привязать. Но имейте в виду, что после настройки домена старый адрес (github.io) будет перенаправлять на новый.

4.2 Введение в Jekyll: генератор статических сайтов

Jekyll – это инструмент, который превращает простые текстовые файлы (например, в формате Markdown) в готовый HTML-сайт. GitHub Pages поддерживает Jekyll «из коробки»: если вы положите в репозиторий файлы с расширением .md и специальную структуру, GitHub автоматически соберёт из них сайт.

Базовая структура Jekyll-сайта

my-site/
├── _config.yml          # главный файл настроек
├── index.md             # главная страница (можно Markdown)
├── about.md             # другая страница
├── _layouts/            # папка с шаблонами
│   └── default.html
└── _posts/              # папка для записей блога
    └── 2025-01-01-hello-world.md

Пример _config.yml (минимальный)

title: Мой блог
description: Интересные заметки
theme: minima   # встроенная тема GitHub Pages

Jekyll автоматически подставит заголовок и описание в HTML-страницы.

Создание первого поста

Создайте файл _posts/2025-04-01-first-post.md с содержимым:

---
layout: post
title: "Мой первый пост"
date: 2025-04-01
---
Привет! Это мой первый пост на Jekyll. GitHub Pages сам превратит его в красивую HTML-страницу.

Закоммитьте и запушите – сайт обновится, и пост появится в блоге (если тема поддерживает отображение постов).

📌 Важно: GitHub Pages автоматически запускает сборщик Jekyll, если в корне есть файл _config.yml. Если вы не хотите использовать Jekyll, просто не создавайте этот файл – GitHub будет отдавать файлы как есть.

4.3 Выбор и установка темы

Темы Jekyll – это готовые наборы стилей и макетов. GitHub Pages предлагает несколько встроенных тем (minima, cayman, hacker и др.). Чтобы выбрать тему, достаточно указать её в _config.yml:

theme: jekyll-theme-cayman

Полный список тем: https://pages.github.com/themes/

Вы также можете использовать сторонние темы, скачав их файлы в репозиторий (но это сложнее). Для начала используйте встроенные.

4.4 Автоматическая сборка с помощью GitHub Actions (продвинутый уровень)

GitHub Actions – это система CI/CD (непрерывная интеграция и доставка). Она позволяет автоматически выполнять скрипты при каждом push, например, собирать проект с помощью npm, запускать тесты или минифицировать CSS.

Для статического сайта на HTML/CSS/JS Actions не обязательны, но они полезны, если вы используете фреймворки (React, Vue) и хотите автоматически собирать приложение перед публикацией.

Пример простого workflow для сборки React-приложения

Создайте в репозитории папку .github/workflows/deploy.yml с содержимым:

name: Deploy to GitHub Pages

on:
  push:
    branches: [ "main" ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm install
      - run: npm run build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./build

Этот workflow при каждом push в ветку main установит зависимости, соберёт проект и выложит результат в ветку gh-pages (которую затем можно использовать как источник Pages).

⚠️ Для начинающих: GitHub Actions – мощный, но сложный инструмент. Не торопитесь его осваивать, пока не уверенно чувствуете себя с Git и основами веб-разработки.

4.5 Продвинутые настройки: файл .gitignore и секреты

Файл .gitignore указывает Git, какие файлы не нужно отслеживать (например, папку node_modules, временные файлы). Пример для Node.js проекта:

node_modules/
.DS_Store
.env

Секреты (secrets) – это переменные окружения, которые можно использовать в GitHub Actions. Они настраиваются в Settings → Secrets and variables → Actions. Например, токен для деплоя на внешний хостинг.

📝 Задание для самопроверки
1. Если у вас есть домен, попробуйте привязать его к вашему GitHub Pages сайту (следуя инструкции DNS).
2. Создайте в репозитории файл _config.yml с любой темой (например, theme: jekyll-theme-cayman). Посмотрите, как изменится дизайн сайта.
3. Напишите один пост в формате Markdown в папке _posts и проверьте, как он отображается.
4. (Для продвинутых) Настройте простой GitHub Actions workflow для автоматической проверки HTML-кода (например, с помощью html-proofer).
5. Если не используете Jekyll, убедитесь, что файл .nojekyll (пустой) лежит в корне – он отключает обработку Jekyll.

4.6 Часто задаваемые вопросы

  • Можно ли использовать свой домен без HTTPS? GitHub Pages автоматически выдаёт бесплатный HTTPS-сертификат через Let's Encrypt, если домен правильно настроен. Это безопасно и не требует дополнительных действий.
  • Как скрыть .github.io из адреса? Только через собственный домен. Пользовательский сайт (ваше_имя.github.io) не может быть «скрыт».
  • Почему после добавления _config.yml мои CSS и JS перестали работать? Jekyll обрабатывает файлы и может игнорировать некоторые папки. Убедитесь, что ваши статические файлы лежат в корне или настройте _config.yml соответствующим образом.
  • Что такое .nojekyll? Пустой файл .nojekyll в корне репозитория говорит GitHub не обрабатывать сайт через Jekyll. Это полезно, если вы хотите использовать собственные инструменты сборки.
