Модуль 4: Подключение устройств – клавиатура, мышь, порты и диски

✍️ Время чтения: ~20 минут | 🎯 Цель: научиться подключать и настраивать клавиатуру, мышь, последовательные порты, дисководы и жёсткие диски в XML-конфигурации PCx86

В предыдущем модуле мы создали базовую конфигурацию с процессором, памятью и видеокартой. Но настоящий компьютер не может работать без устройств ввода-вывода. В этом модуле мы подключим клавиатуру, мышь, последовательные порты, дисководы для дискет и жёсткие диски. Вы также узнаете, как управлять этими устройствами через графический интерфейс и добавлять панель управления с отладчиком.

💡 Зачем подключать устройства?
Эмулятор должен максимально точно воспроизводить поведение реального железа. Клавиатура и мышь нужны для взаимодействия с пользователем, последовательные порты – для обмена данными с другими устройствами (например, с терминалом или отладчиком), а диски – для хранения операционной системы и программ. Без них эмулятор будет просто «голым» процессором с памятью.

4.1. Клавиатура (<keyboard>)

Клавиатура в PCx86 эмулирует раскладки IBM PC. Она преобразует события JavaScript (нажатия клавиш на физической клавиатуре) в скан-коды, понятные эмулируемой системе.

Основные атрибуты тега <keyboard>:

  • model – определяет раскладку клавиатуры. Доступные варианты:
    • us83 – клавиатура IBM PC/XT на 83 клавиши (по умолчанию).
    • us84 – клавиатура IBM PC/AT на 84 клавиши.
    • us101 – клавиатура на 101 клавишу (современная).
  • bindings – позволяет привязать кнопки интерфейса к определённым комбинациям клавиш (например, ctrl-alt-del для перезагрузки).

Пример минимальной конфигурации клавиатуры:

<keyboard model="us84" />

Более сложный пример с кнопками управления (обычно такие кнопки добавляют в панель управления):

<keyboard model="us84">
    <control type="button" binding="ctrl-alt-del">Ctrl-Alt-Del</control>
    <control type="button" binding="keyboard">Soft Keyboard</control>
</keyboard>

Первая кнопка при нажатии отправит в эмулятор комбинацию Ctrl+Alt+Del (перезагрузка). Вторая – вызовет экранную («soft») клавиатуру, что удобно на мобильных устройствах.

📌 Важное примечание из документации PCx86: Компонент Video автоматически привязывается к событиям клавиатуры на своём canvas-элементе. Поэтому для обычного ввода с клавиатуры не требуется никаких дополнительных действий. Кнопки с привязками (binding) нужны только для эмуляции системных комбинаций (например, Ctrl+Alt+Del) или для отображения экранной клавиатуры.

4.2. Мышь (<mouse>)

PCx86 поддерживает три типа мыши: последовательную (serial), шинную (bus) и InPort. На практике чаще всего используется последовательная мышь, подключённая к COM-порту.

Атрибуты компонента <mouse>:

  • type – тип мыши: "serial", "bus" или "inport" (по умолчанию "serial").
  • binding – имя последовательного порта (компонента <serial>), к которому подключается мышь. Используется только для type="serial".
  • adapter – для мыши InPort: 1 (основной) или 2 (вторичный).
  • scaleMouse – коэффициент масштабирования координат (по умолчанию 0.5).

Пример подключения последовательной мыши к первому COM-порту:

<mouse type="serial" binding="com1" />

Обратите внимание: для работы мыши необходимо также определить последовательный порт (<serial>), который будет описан ниже.

4.3. Последовательные порты (<serial>)

Компонент <serial> эмулирует стандартные COM-порты IBM PC. Они используются для подключения мыши, модема, отладочной консоли или даже связи с другим эмулятором (например, с VT100-терминалом).

Атрибуты:

  • id – уникальный идентификатор порта (например, "com1", "com2").
  • adapter – номер адаптера (1 или 2), который определяет базовый адрес порта (COM1: 0x3F8, COM2: 0x2F8).
  • binding – специальное назначение порта (например, "print" для вывода данных в окно браузера).
  • tabsize, char – параметры форматирования при выводе на печать.

Типичная конфигурация двух последовательных портов (как в машине IBM PC AT 5170):

<serial id="com1" adapter="1" />
<serial id="com2" adapter="2" binding="print" tabsize="8" char="·" />

В этом примере com1 используется для мыши или модема, а com2 привязан к виртуальному принтеру – всё, что отправляется на COM2, будет выводиться в специальном окне браузера (полезно для отладки).

💡 Углублённо: На сайте PCjs есть демонстрация, где IBM PC AT соединён через COM2 с эмулятором терминала VT100. Это позволяет «общаться» с другой виртуальной машиной. Такие сценарии выходят за рамки нашего модуля, но знание о них расширяет возможности.

4.4. Чипсет (<chipset>)

Чипсет – это «клей», который связывает процессор, память и периферию. В PCx86 чипсет определяет модель системной платы и, что важно для нас, количество дисководов.

Атрибуты:

  • model – модель чипсета. Для IBM PC AT используется "5170".
  • floppies – массив поддерживаемых типов дисководов. Например, [1440, 1200] означает два дисковода: на 1.44 МБ и на 1.2 МБ.

Пример для IBM PC AT с двумя дисководами:

<chipset id="chipset" model="5170" floppies="[1440,1200]" />

Этот параметр важен, потому что контроллер дисководов (FDC) использует настройки чипсета для определения количества доступных приводов.

4.5. Контроллер гибких дисков (<fdc>)

FDC (Floppy Disk Controller) управляет дисководами для дискет. Он может автоматически монтировать образы при загрузке, а также предоставляет пользовательский интерфейс для выбора и загрузки дискет.

Атрибуты и вложенные элементы:

  • autoMount – объект, определяющий, какие образы автоматически вставляются в дисководы при старте.
  • ref – ссылка на внешний XML-файл с описанием контроллера (обычно не требуется, если вы пишете конфигурацию с нуля).
  • diskettes – список JSON-файлов с описанием доступных образов дискет (используется для выпадающего списка).
  • server – базовый URL для загрузки образов.

Внутри тега <fdc> можно размещать элементы управления:

  • <control type="list" binding="listDrives"> – выпадающий список дисководов (A:, B:).
  • <control type="list" binding="listDisks"> – список доступных образов дискет (заполняется вручную).
  • <control type="button" binding="loadDisk">Load</control> – кнопка загрузки выбранного образа.
  • <control type="button" binding="saveDisk">Save</control> – кнопка сохранения текущего образа (с изменениями).
  • <control type="file" binding="mountDisk"> – поле для загрузки локального файла-образа.

Пример полной конфигурации FDC (взят из реальной машины):

<fdc id="fdcNEC" autoMount="{A:{name:'PC DOS 2.00 (Disk 1)'}, B:{name:'PC DOS 2.00 (Disk 2)'}}"
     diskettes="/machines/pcx86/diskettes.json,/disks/private/pcx86/diskettes.json"
     server="/disks/diskettes" sortBy="name" pos="left" padLeft="8px" padBottom="8px">
    <control type="list" binding="listDrives"></control>
    <control type="list" binding="listDisks">
        <disk id="PCDOS200-DISK1" name="PC DOS 2.00 (Disk 1)"></disk>
        <disk id="PCDOS200-DISK2" name="PC DOS 2.00 (Disk 2)"></disk>
    </control>
    <control type="button" binding="loadDisk">Load</control>
    <control type="button" binding="saveDisk" padRight="8px">Save</control>
    <control type="description" binding="descDisk" padRight="8px"></control>
    <control type="file" binding="mountDisk"></control>
</fdc>

Обратите внимание: список listDisks заполняется вручную тегами <disk>. Это позволяет предлагать пользователю только те образы, которые действительно нужны для данной машины.

4.6. Контроллер жёсткого диска (<hdc>)

HDC (Hard Disk Controller) управляет жёсткими дисками. В PCx86 поддерживаются два типа контроллеров: XT (для старых PC) и AT (для PC AT и новее).

Атрибуты:

  • type – тип контроллера: "XT" или "AT".
  • drives – массив объектов, каждый из которых описывает один жёсткий диск. Каждый объект содержит:
    • name – отображаемое имя диска.
    • type – тип диска (номер геометрии). Для XT обычно 3 (10 МБ), для AT – 1 (10 МБ), 2 (20 МБ) и т.д.
    • path – путь к JSON-файлу с образом диска.
  • ref – ссылка на внешний XML с конфигурацией HDC (может быть полезна для переиспользования).

Типы жёстких дисков для PC XT:

  • Тип 0: 306 цилиндров, 2 головки, 17 секторов (5.3 МБ).
  • Тип 1: 375 цилиндров, 8 головок, 17 секторов (26.1 МБ).
  • Тип 2: 306 цилиндров, 6 головок, 17 секторов (15.9 МБ).
  • Тип 3: 306 цилиндров, 4 головки, 17 секторов (10.6 МБ).

Для PC AT используются другие номера типов, хотя геометрия может совпадать. Например, диск на 10 МБ в AT имеет тип 1 (306 цилиндров, 4 головки, 17 секторов). В документации PCjs рекомендуется для AT использовать тип 2 для 20-мегабайтных дисков и тип 1 для 10-мегабайтных.

Пример конфигурации HDC для PC AT с одним жёстким диском на 20 МБ:

<hdc id="hdcAT" type="AT" drives="[{name:'20Mb Hard Disk', type:2, path:'/harddisks/pcx86/20mb/IBMOS210-EGA.json'}]">
    <control type="button" binding="saveHD0">Save HD</control>
</hdc>

Кнопка Save HD позволяет пользователю сохранить изменённый образ жёсткого диска на свой компьютер.

📌 Важно: Если вы переносите образ диска с сайта PCjs, убедитесь, что путь указан правильно. Для локальных файлов используйте относительные пути (например, path="my-harddisk.json"), если файл лежит в той же папке, что и XML-конфигурация.

4.7. Панель управления (<panel>)

Панель управления – это контейнер для кнопок, индикаторов и полей ввода, которые управляют эмулятором (запуск, остановка, сброс, отладка). Обычно она используется вместе с отладчиком.

Атрибутов у панели нет, но внутри неё можно размещать различные элементы управления (<control>).

Пример простой панели с кнопками управления CPU:

<panel>
    <control type="button" binding="run">Run</control>
    <control type="button" binding="step">Step</control>
    <control type="button" binding="reset">Reset</control>
</panel>

Для работы отладчика панель управления должна быть включена в конфигурацию, а также должен быть добавлен компонент <debugger>. Подробнее об отладчике – в следующем разделе.

4.8. Отладчик (<debugger>)

Отладчик – мощный инструмент для разработчиков эмуляторов. Он позволяет останавливать выполнение, просматривать содержимое памяти, регистров, устанавливать точки останова.

Атрибуты отладчика:

  • messages – категории сообщений, которые будут выводиться в консоль. Можно указать "fault" (только ошибки), "info" и т.д.
  • commands – путь к файлу с командами (необязательно).

Чтобы включить отладчик, необходимо:

  1. Добавить в XML компоненты <debugger> и <panel>.
  2. Добавить на панель управления элементы для управления отладчиком: кнопки Run, Step, Reset, поле ввода команд и т.д.
  3. В HTML-странице загружать не обычный pcx86.js, а pcx86-dbg.js (отладочную версию).

Пример минимальной конфигурации с отладчиком:

<debugger id="debugger" messages="fault"></debugger>
<panel>
    <control type="text" binding="debugInput" width="400px"/>
    <control type="button" binding="debugEnter">Enter</control>
    <control type="button" binding="run">Run</control>
    <control type="button" binding="step">Step</control>
    <control type="button" binding="reset">Reset</control>
</panel>

Поле ввода debugInput позволяет вводить команды отладчика (например, d 0x400 для просмотра памяти). Кнопка debugEnter отправляет команду.

⚠️ Внимание: В большинстве готовых конфигураций отладчик выключен (атрибут messages="none") или вовсе отсутствует. Для обычного пользователя он не нужен. Но если вы создаёте собственный эмулятор и хотите отлаживать загрузку ОС, отладчик незаменим.

4.9. Полный пример конфигурации (IBM PC AT с периферией)

Соберём всё вместе и создадим XML для машины IBM PC AT 5170 с 2 МБ RAM, EGA-видео, двумя последовательными портами, мышью, двумя дисководами и жёстким диском:

<machine id="ibm5170" type="pcx86" border="1" pos="center">
    <name>IBM PC AT (8Mhz), 2Mb RAM, 20Mb HDD, IBM EGA</name>
    <computer id="at-ega-2048k" name="IBM PC AT" buswidth="24" />
    <cpu id="cpu286" model="80286" cycles="8000000" autostart="true" />
    <ram id="ramLow" addr="0x00000" size="0xa0000" test="false" />
    <ram id="ramExt" addr="0x100000" size="0x160000" />
    <rom id="romBIOS" addr="0xf0000" size="0x10000" alias="0xff0000" file="ATBIOS-REV3.json5" />
    <video model="ega" memory="0x20000" screenWidth="1280" screenHeight="700">
        <rom id="romEGA" addr="0xc0000" size="0x4000" file="IBM-EGA.json5" notify="videoEGA" />
    </video>
    <chipset id="chipset" model="5170" floppies="[1440,1200]" />
    <fdc autoMount="{A:{name:'MS-DOS 6.22', path:'msdos622.img'}}">
        <control type="list" binding="listDrives"></control>
        <control type="list" binding="listDisks">
            <disk id="MSDOS622" name="MS-DOS 6.22"></disk>
            <disk id="UTILS" name="Utilities"></disk>
        </control>
        <control type="button" binding="loadDisk">Load</control>
        <control type="button" binding="saveDisk">Save</control>
    </fdc>
    <hdc id="hdcAT" type="AT" drives="[{name:'20Mb Hard Disk', type:2, path:'my-hdd.json'}]">
        <control type="button" binding="saveHD0">Save HD</control>
    </hdc>
    <keyboard model="us84" />
    <mouse type="serial" binding="com1" />
    <serial id="com1" adapter="1" />
    <serial id="com2" adapter="2" binding="print" tabsize="8" char="·" />
    <panel>
        <control type="button" binding="ctrl-alt-del">Ctrl-Alt-Del</control>
    </panel>
</machine>

Сохраните этот код как my-full-machine.xml, поместите рядом все необходимые файлы (образы BIOS, дисков) и запустите через HTML-шаблон из модуля 2. Вы получите полностью работоспособную виртуальную машину с графическим интерфейсом для управления дисководами и кнопкой перезагрузки.

📝 Задание для самопроверки
1. Добавьте в свою конфигурацию из модуля 3 клавиатуру с раскладкой us101 и мышь, подключённую к COM1.
2. Настройте чипсет для поддержки двух дисководов (на 1.44 МБ и 720 КБ).
3. Создайте панель управления с кнопками «Run», «Step» и «Reset».
4. Подключите отладчик (включите вывод сообщений о неисполненных инструкциях).
5. Убедитесь, что при загрузке эмулятора в дисковод A автоматически монтируется образ дискеты с MS-DOS.

4.10. Где брать образы дисков и BIOS?

Образы BIOS и дискет в формате JSON можно скачать с сайта PCjs или из репозитория на GitHub. Например:

  • BIOS для IBM PC AT: https://www.pcjs.org/machines/pcx86/ibm/5170/rom/bios/1985-11-15/ATBIOS-REV3.json5
  • Образ MS-DOS 6.22: можно найти в архиве сайта PCjs или создать самостоятельно с помощью утилиты DiskDump.

Для учебных целей допустимо использовать прямые ссылки на файлы с сайта PCjs (как в примере выше), но для полноценного сайта лучше скачать их и разместить локально.
